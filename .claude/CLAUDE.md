## Project

Browser-based realtime web-audio synthesizer. POC milestone: a sine wave generated by a C++ DSP module compiled to wasm, run inside an `AudioWorkletProcessor`, audible end-to-end. Path 3 (AudioWorklet + Wasm) was chosen from the original three-path tradeoff; pivoted Rust‚ÜíC++/Emscripten on 2026-05-07 so existing JUCE C++ DSP code can be ported forward without language rewrite.

The user has standing read permission on the external reference repo at `C:\REPOS\PLUGIN_PROJECTS\EXTERNAL\web-synth` (Ameobea's `web-synth`). Mine it for AudioWorklet bootstrap and raw `wasm` instantiation patterns. Do **not** pull in its patch-network model, ViewContext system, jantix store, litegraph editor, Faust/Soul integration, or backend.

## Architecture (big picture)

**Two-repo split.** Pure DSP lives in a standalone library `RD_DSP`, consumed:
- by JUCE plugins natively, and
- by WEB_SYNTH via git submodule at `SUBMODULES/RD_DSP/`, cross-compiled to wasm.

WEB_SYNTH itself is web-glue + C-ABI shim layer: AudioWorklet, HTML, the `extern "C"` wasm exports, and a thin C++ class per module that owns `rd_dsp::` instances and adapts per-sample DSP into block-rate `process()` calls.

**Paired processor pattern.** For each module, a C++ class lives in `ENGINE/<MODULE>/` and a JS `AudioWorkletProcessor` lives in `PUBLIC/` under the same conceptual name. C++ class holds DSP objects as `m`-prefixed members; JS side handles the audio-thread boundary and `MessagePort` traffic.

**Wasm shape.** Standalone wasm ‚Äî `-sSTANDALONE_WASM=1 --no-entry`, no JS glue from emcc. The worklet `WebAssembly.instantiate`s the bytes manually (main thread fetches, posts via `MessagePort` since worklets cannot `fetch`). Audio samples cross the boundary as a `Float32Array` view over wasm linear memory at a fixed exported buffer pointer ‚Äî zero-copy, no per-block allocation.

**Build layout.** Thin root `CMakeLists.txt` adds each `ENGINE/<MODULE>/` subdir. Each module's `CMakeLists.txt` defines an `add_executable` wasm target with `EMSCRIPTEN`-guarded link options (`STANDALONE_WASM`, `EXPORTED_FUNCTIONS`, `--no-entry`) and a post-build step that copies the artifact to `PUBLIC/<MODULE>.wasm`. Per-module source lists live in generated, committed `CMAKE/<MODULE>_SOURCES.cmake` files ‚Äî those files are the contract between the filesystem and CMake, regenerated by `SCRIPTS/regenSource.py` on add/remove. Ninja generator. Out-of-tree build dir `BUILD/` (gitignored).

For deeper detail, see:
- `.claude/plan.md` ‚Äî incremental POC plan with per-step status, notes on what is done vs. pending. Source of truth for "what's next".
- `.claude/rd_dsp_integration_plan.md` ‚Äî RD_DSP cross-repo contract + incremental wiring plan: linkage shape, wasm-safety rules, block-rate API conventions, step-by-step status.

## Commands

- Build the current wasm module: `python SCRIPTS/build_synth.py` from repo root. Add `--clean` to wipe `BUILD/`. Artifact lands at `PUBLIC/synth.wasm`.
- Regenerate source lists after adding or removing a file under `ENGINE/<MODULE>/`: runs automatically as the first step of `build_synth.py`. Standalone invocation: `python SCRIPTS/regenSource.py`. Generated `CMAKE/<MODULE>_SOURCES.cmake` files are committed.
- No dev server, lint, typecheck, or test commands yet (plan increments 4‚Äì8 not started). Document here when added.

**Toolchain:** emsdk at `C:\emsdk` (emcc 5.0.7), Ninja 1.13.2 (`winget install Ninja-build.Ninja`), CMake 3.30.2.

**Windows emsdk gotchas:** `emsdk_env.sh` shells to `python` and hits the Windows Store stub ‚Äî `export EMSDK_PYTHON=/c/Users/.../python.exe` before sourcing. Windows emsdk ships only `emcc.bat`/`.ps1`/`.py` (no extensionless wrapper); Git Bash will not auto-resolve `.bat`. Build script invokes `python emcc.py` directly to avoid PATH mutation and shim layers.

## Working directory layout

- `ENGINE/<MODULE>/` ‚Äî C++ source for one wasm module (currently `SYNTH/`).
- `SUBMODULES/RD_DSP/` ‚Äî pure DSP submodule, built into the wasm via `add_subdirectory` + `target_link_libraries`.
- `PUBLIC/SUBMODULES/RECLUSE_UI/` ‚Äî shared design-system submodule (tokens + custom-element components). Browser-served directly from `PUBLIC/`. Consumed by `index.html` (`tokens/*.css`) and `rd-pulsar.js` shadow root (CSS vars inherit through shadow boundary). Bump pointer: `git submodule update --remote PUBLIC/SUBMODULES/RECLUSE_UI`. Use `--color-*` token vars over hex literals in component styles.
- `CMAKE/<MODULE>_SOURCES.cmake` ‚Äî generated, committed source lists.
- `SCRIPTS/` ‚Äî Python build/dev helpers.
- `PUBLIC/` ‚Äî static web assets + emitted `.wasm`. Must be served over HTTP/HTTPS (AudioWorklet cannot load from `file://`).
- `BUILD/` ‚Äî CMake/Ninja out-of-tree output. Gitignored.
- `CMakeLists.txt` ‚Äî thin root, just `add_subdirectory`s.
- `WEB_SYNTH.code-workspace` ‚Äî VS Code workspace.
- `.claude/` ‚Äî `CLAUDE.md` (this file), `plan.md`, `rd_dsp_integration.md`, `settings.local.json`.
